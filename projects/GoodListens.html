GoodListens
Make sense of the noise.

A group project to help podcast listeners track what they listen to and discover new content they love.

UX Designer / React Native Developer

React Native / Firestore

Dec. 2019

GoodListens is designed for avid podcast listeners who want to track what they and their friends listen to each day. Just like books, today there are far too many podcasts being produced for one person to consume them all. By tracking the podcasts you listen to you can create a log to reference later, and by following friends you can see what podcasts they like and find new favorites! The app is a simple and powerful tool to hone in on the podcast content you love.

Development Process

We chose to implement the user interface with React Native and stored all the necessary data in a noSQL database powered by Google’s Firebase. My three-person group had less than three weeks to design, implement, and present this mobile app for our final project. This forced us to make quick decisions and prioritize only the necessary features that would give users the experience we wanted.

With our feature list in hand we used Adobe XD to collaboratively design rough mockups to agree on the screens our app would need.

The mockups helped us sketch out an initial structure for our database.

Database sketch here

My group used Github to manage our codebase and project tasks. We created seven milestones that needed to be met for the project to be a success and set due dates for each of them. As we worked toward a milestone we broke them into smaller, individual tasks, and opened Github issues to assign one or more people to implement the feature.

I acted as the team’s Buildmaster. I outlined the process we used for branching and merging, and helped people with their git errors.

While our project had to move quickly, our code review process was thorough. When anyone finished their work on a new feature, they notified the others in the group that their code was ready for review. We would test their branch locally and would only accept the new code if the other two group members approved. This ensured we didn’t waste any time taking steps backward when new features were added.

Results

At the end of the project our group was able to create a React Native mobile app that was deployed using Expo. We managed to implement all but one of the features we prioritized.

Reflection

This project offered a great amount of learning moments. I had professional experience collaboratively writing code in a mature codebase, but I had never started such a group project from scratch. We saw how slow progress could be when everyone was making changes to the same few files, and it took great teamwork to make any headway.

Our tight deadline also showed me what we could accomplish when time is the biggest constraint. There was no time to endlessly debate how a feature would be implemented. We had to quickly discuss workplans and trust that we would all carry our weight, and it resulted in an app we were proud to demo at the end of the project.
